1Webhook entry — routes/whatsapp.js

Inbound dispatch

  • verifyMetaSignature at L24 — rejects requests without valid X-Hub-Signature-256.
  • For each messages[] item: branch on type (text · image · document · interactive · audio · etc.).
  • Interactive buttons map IDs (doc:yes, confirm:yes, burst:done…) into synthetic user text before runTurn.

Why text and media fork early

Media must be downloaded before entering runTurn because the lock is per-session and the binary fetch (Meta CDN auth) is too slow to hold under it. side-effect if fetchMedia returns null we now early-return with a citizen-facing notice just added.

2Media fetch — routes/whatsapp.js:207

Steps

  1. GET /v18.0/{media_id} → returns short-lived signed URL.
  2. GET that URL with bearer → binary.
  3. Persist to ./data/uploads/{session_id}/{ts_rand}.{ext} via extForMime (L198).
  4. Return { url:'/uploads/...', mime, name }.

Failure modes (relevant to current bug)

  • silent-null unsupported MIME (e.g. application/octet-stream) → returns null → bot then calls runTurn with no attachment, and the LLM hallucinates a save.
  • stale URL Meta CDN URL expires in ~5 min — fetch must complete inside the same handler.
  • disk if ./data/uploads isn't on a Render persistent disk, files are lost on redeploy.

3runTurn / session lock — lib/agent.js:411

Why the lock matters

withSessionLock at L397 serializes turns per session_id. Without it, two near-simultaneous WhatsApp messages would both read the same state.collected, both write, and the second clobbers the first.

Inflight counter

bumpInflightFiles (L94) keeps a per-session count of media-bearing turns currently waiting on the lock. drainBurst reads this at L136 to decide whether to fire — see §4.

4Burst aggregator — lib/agent.js:109 / :130

Citizens send 3–4 documents in quick succession. Without aggregation we'd reply once per file ("got page 1", "got page 2", …). The aggregator buffers replies until the citizen stops sending for BURST_QUIET_MS (1200ms).

Inflight gate (L136). If a previous turn is still inside the session lock, drainBurst bails out and re-arms — preventing the "sent reply too early, before the next file's turn finished" race that produced duplicate "✅ استلمت" lines.

Deterministic overrides

  • welcome first turn (state idle, no recent bot) → fixed welcome copy, skips LLM.
  • start_submission when LLM calls this tool, the human-facing reply is built from the tool result, not from the LLM's own text — kills the hallucinated-fee/wrong-name class.
  • first_doc after start_submission, prompt with firstDocPrompt (L635) using catalog data.

Prompt cache

  • system block carries cache_control:{type:'ephemeral'}.
  • The last tool_result also gets a cache breakpoint — Anthropic 5-min TTL covers the typical citizen reply window.
  • Visible in lib/llm.js — Anthropic when ANTHROPIC_API_KEY set, Qwen otherwise.

The system prompt explicitly forbids the LLM from writing "اكتب نعم/لا" or "اكتب تم" — those phrases are the signal we missed an auto-attach, not a UX feature. See SYSTEM_V2 § Channel UX in lib/agent.js.

8Hallucination guard — lib/agent.js:2552

Post-LLM text is regex-scanned for "save-claim" phrases. If matched but no record_* tool fired this turn, we either:

  • Mode A — no record at all: strip the claim, append "أرسل المستند الآن" (send now).
  • Mode B — record fired but for a different slot: rewrite the slot label to match what the tool actually wrote.
// current pattern (with "استلمت" extension just added):
const HALLUCINATED_SAVE_RE =
  /✅\s*(?:حفظت|استلمت|تم\s*الحفظ|تم\s*التسجيل|تم\s*الاستلام|saved|received|recorded|stored|got\s+your)
    |(?:^|\s)(?:حُفِظ(?:ت|ا)?|سجّلت|سُجّل|received\s+your)(?=\s|$|[:.،])/ix
The Arabic-aware lookahead (?=\s|$|[:.،]) is required — JS \b doesn't fire on Arabic letters, so a naive boundary lets the verb leak past the guard.

10Live bugs we're investigating

A · "✅ استلمت البطاقة المدنية" but state.collected is empty

Trace #1184: bot text claims a save; state.collected.civil_id is still ⬜. No record_document tool call in the round.

Root cause: hallucination guard regex didn't match استلمت (only matched حفظت/تم الحفظ).

Fix: extended HALLUCINATED_SAVE_RE at lib/agent.js:2552 to include استلمت, تم الاستلام, received your.

Verify: re-run trace, confirm guard rewrites or strips the bot reply when no record_* tool fired.

B · file 1 has no media_url in trace

Trace #1183: attachment present in webhook payload but fetchMedia returned null. The bot then proceeded into runTurn with no attachment and the LLM made up a confirmation.

Root cause: webhook used to silently drop unsupported MIMEs.

Fix: in routes/whatsapp.js, when fetchMedia returns null, send the citizen "⚠️ لم أستطع استلام الملف…" and skip runTurn entirely — no chance for the LLM to invent a save.

Verify: send a corrupted/oversized file via WA, expect the new notice text and no record_* tool call in the trace.

C · "اكتب نعم/لا" / "اكتب تم" still leaking

System prompt forbids these phrases; LLM occasionally produces them anyway.

Mitigation in place: looksLikeYesNoAsk heuristic auto-attaches buttons even if the prompt fails — see §7. Recheck: ensure the heuristic catches Arabic confirm phrasing without depending on \b.

★Where to look next

If reply is wrong
Read lib/agent.js at the runAgentV2 finalize block — that's where the LLM text is sanitized, then the hallucination guard, then button attach. Most "weird reply" bugs land in one of those three.
If a file vanished
Walk: webhook receive → fetchMedia → upload to disk → runTurn attachment → handleCollecting / V2 record_document → state.collected[code]. Compare against /api/debug/trace.
If the bot replied twice
Burst aggregator. Check inflightFilesFor(session_id) at the moment of the timer fire (L136). If >0 the timer should have re-armed — if it didn't, that's the bug.
If a button didn't appear
Check finalize order: deterministic override → text → guard → button heuristic. If the heuristic regex doesn't match, no button. lib/agent.js:2698.
